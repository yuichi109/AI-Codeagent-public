## 基本方針：通常モードとは完全に分離

**チャット画面に「🤖 マルチエージェント」トグルボタンを設置し、明示的にONにした場合のみ動作する。**

- デフォルトはOFF（通常の単一エージェントモード）
- 重要なタスク・品質を特に求める場合にだけ使う
- 理由：複数モデルを複数ターン動かすためトークン消費が大幅に増える
- ONにしたまま雑談・軽いタスクに使うのは非推奨

---

## なぜこの機能が必要か

エンジニア仲間が同様の仕組みで、AIエージェントに高効率・高品質な成果物を作らせている実績がある。

単一のAIが全部やるより、**役割を分担した複数のAIが協調**することで：
- 各エージェントが自分の専門に集中できる
- 複数モデル（GPT-4.1-mini / GPT-4.1 / gpt-5-mini 等）を役割ごとに割り当てられる
- 並列実行により処理速度が上がる
- 一つのエージェントの見落としを別のエージェントが補える

## システム概要

```
ユーザーがチャットで指令
  └─ ディスパッチャー（タスク分解・割り当て）
       ├─ 設計エージェント
       │    └─ design.md を書く
       ├─ コーディングエージェント
       │    └─ code.py + how-to-use.md を書く
       ├─ ドキュメントエージェント
       │    └─ README等を書く
       └─ デバッグ/テストエージェント
            └─ test-result.md を書く

各エージェントは互いのドキュメントを読んで自分の仕事をする
進捗をステータスファイルに書き込む
閾値 or 納期を超えたら中間報告
```

## 核心的な設計方針

### エージェントは完全に独立・専門特化する

これはこのシステムの最重要原則。

- 各エージェントは**自分の責任範囲だけを実行する**
- 他のエージェントの出力を読むことはあっても、**修正・改善・手伝いは一切しない**
- 「ついでにここも直しておこう」は禁止。スコープ外は無視する
- エージェント間の仕事の取り合い・割り込みはない

**なぜこれが重要か：**  
一つのAIに全部任せると、どれほど優秀なモデルでも勝手に範囲を広げて変な方向に進むことがある。  
役割を厳格に切ることで各AIが専門に集中し、結果として全体の精度が上がる。  
また、何かがおかしくなったときに「どのエージェントの出力が原因か」を特定しやすくなる。

**実装上の意味：**
- 各エージェントのシステムプロンプトに「あなたの仕事はXのみ。それ以外は出力しないこと」を明記
- 読んで良いファイル・書いて良いファイルをエージェントごとに定義し、それ以外には触れさせない
- ディスパッチャーだけが全体を把握し、流れを制御する

---

### 通信手段はファイルのみ・会話は一切しない

**エージェント同士は会話しない。ドキュメントだけが唯一の成果・通信手段。**

- エージェントが残せるのはファイルだけ（コード・設計書・テスト結果・使い方等）
- 他のエージェントへのメッセージ送信・返答・質問は存在しない
- 他のエージェントの成果ファイルを読んで自分の仕事に活かすことはあるが、それだけ
- コーディングエージェントはコード本体と「このコードの使い方・意図」を両方ファイルとして残す
- 会話がないからこそ干渉が起きない・スコープが崩れない

**なぜ会話させないか（実体験からの教訓）：**

多くの実装者が「上司AI × 専任AI 8人」のような会話型マルチエージェントを試した結果、
無駄な遠回り・意味のないループ・トークン爆発が頻発することが分かっている。
AIは会話の中で「確認」「相談」「訂正」を繰り返し、本来の仕事より会話コストが膨らむ。

**成果物ファイルだけでやり取りする方が無駄がなく、結果として品質も上がる。**

実装時の注意：「エージェント間でやり取りさせた方が賢くなるのでは？」という誘惑に負けないこと。
それは罠であり、この設計の根幹を壊す。

この原則により：
- アーキテクチャがシンプルになる（リアルタイム通信の仕組みが不要）
- 全エージェントの仕事がファイルとして残るため、後から何をやったか追跡できる
- 人間がいつでも途中経過を確認・介入できる

### ループと完了判定・納期管理

全体の進捗を把握しているのはディスパッチャーだけなので、**ループ管理・催促・中間報告はすべてディスパッチャーが担う**。

**通常の完了フロー：**
- 各エージェントが完了したら status.md に完了を記録
- ディスパッチャーが全員の完了を確認したら最終報告をユーザーに返す
- 品質閾値を超えなければ該当エージェントを再実行してループ

**納期超過フロー：**
- ディスパッチャーが定期的に status.md を確認する
- 設定した制限時間を超えてもエージェントが完了していない場合、**ディスパッチャーから催促**を送る
  - 催促 = そのエージェントを再度呼び出し「残り時間Xです。現時点の成果を提出してください」と伝える
- 最終納期を過ぎたら完成・未完成にかかわらず**中間報告を生成してユーザーに返す**

**設定可能にするパラメータ：**
- 最大ループ回数
- エージェントごとのタイムアウト（催促トリガー）
- 全体の制限時間（中間報告トリガー）
- 品質閾値（何をもって「完了」とするか）

### モデルの割り当て

役割ごとに異なるプロバイダー・モデルを割り当てられる。  
設定は `/setup` → 「🤖 マルチエージェント設定」セクションで行う（`.multi_agent_config.json` に保存）。

#### 推奨デフォルト設定（バランスプリセット）

| 役割 | 推奨プロバイダー | 推奨モデル | 理由 |
|---|---|---|---|
| ディスパッチャー | Azure AI Foundry | gpt-5.4 | タスク分解の判断精度が全体の品質を左右する |
| 設計 | Azure AI Foundry | gpt-5.4 | アーキテクチャの質がコード全体に影響する |
| コーディング | Azure AI Foundry | gpt-5.4-mini | 速度重視・設計書があれば mini で十分 |
| デバッグ/テスト | Azure AI Foundry | gpt-5.4-mini | 複雑なエラー解釈があるため nano は非推奨・mini が最低ライン |
| セキュリティ | Azure AI Foundry | gpt-5.4 | 脆弱性の見落としは致命的なので高性能必須 |
| ドキュメント | Azure AI Foundry | gpt-5.4-mini | 重要だが high まで使わなくてよい |
| リサーチ | Google Gemini | gemini-2.5-flash | OpenAI 未使用時の現実解・Flash で十分 |
| インフラ | Azure AI Foundry | gpt-5.4 | 環境構築の判断ミスはリカバリーが大変 |

**nanoの使いどころ**: コスト優先プリセットのコーディング・リサーチのみ。デバッグ/テストには使わないこと。

| インフラ | gpt-5.4 | gpt-5.4 | gpt-5.4-mini |

※ プロバイダーを横断した割り当てが可能。Foundry の複数インスタンス（FOUNDRY_2_* 等）も利用できる。

## 実装フェーズ

### Phase 1: 逐次版（まず動かす）
- ディスパッチャーがタスクを分解し、各エージェントを**順番に**呼ぶ
- 現在の `call_responses_api` の発展形として実装可能
- UIは既存のチャット画面で進捗をテキスト表示

難しさ: ★★

### Phase 2: 並列版
- `asyncio.gather` で複数エージェントを同時実行
- UIに複数エージェントの進捗を並列表示（ストリーム）
- SSE or WebSocket での複数ストリーム管理

難しさ: ★★★★

### Phase 3: ループ・品質制御
- デバッグエージェントが品質を評価してループを制御
- 閾値・ループ上限・納期の設定UI
- 中間報告の自動生成

難しさ: ★★★

## 役割一覧（エージェント候補）

タスクの性質によってディスパッチャーが必要な役割を選んで動かす。全員を常に起動するわけではない。

| 役割 | 必須/任意 | 補足 |
|---|---|---|
| ディスパッチャー | 必須 | 常に1名。タスク分解・割り当て・進捗管理・催促・中間報告 |
| リサーチ | タスク次第 | 設計の前に実施。web_search / web_fetch / web_research で複数ページを巡回し調査結果をファイルに書く。Deep Research的な役割。新技術・外部API・ベストプラクティス調査が必要な場合のみ起動 |
| 設計 | 必須 | 高性能モデル1名。アーキテクチャ・既存コードとの整合性。リサーチ結果があれば読んでから設計する |
| コーディング | 必須 | タスク次第で1〜複数。ファイル境界で分割できる場合のみ並列化 |
| インフラ | タスク次第 | 実環境が必要な場合のみ。コンテナ・クラウドリソース構築・動作確認・環境情報をファイルに残す |
| レビュー/デバッグ | 必須 | 実環境でのテスト・品質評価・ループ判定 |
| セキュリティレビュー | タスク次第 | 認証・API・ユーザー入力を扱うコードの場合のみ起動。機能の正しさではなく脆弱性（インジェクション・認証不備等）のみを見る。コーディング完了後・デバッグ前に実施 |
| テスト生成 | 任意 | テストコードを書く専任。デバッグ（動かして確認）とは別スキル。pytestやunittestのテストケース設計・実装を担う |
| ドキュメント | 任意 | changelog・設計書・コメント整備。省略可能なことも |

**インフラ担当が必要な理由：**  
設計AIが「このアーキテクチャで」と決めても、実際に動く環境で検証されなければ空論になる。  
インフラAIが実際にコンテナ・クラウドを構築して動作確認することで、設計とコードが現実の環境で成立することを保証する。

**コーディングを複数名にする条件：**  
触るファイルが完全に別である場合のみ並列化可能。依存関係がある場合はディスパッチャーが逐次実行に切り替える。

## インフラ担当AIの段階設計

### 対象プラットフォームと用途の整理

| 段階 | インフラ担当 | 対象ユーザー |
|---|---|---|
| Windows版 | なし（コード生成・ドキュメントまで） | 手軽にコード作成したい人 |
| WSL版 現在（Stage 1） | ローカル・Dockerのみ | 技術者・自分 |
| WSL版 次の段階（Stage 2） | 登録済みOS環境へリモート接続 | 技術者・自分 |
| WSL版 最終段階（Stage 3） | Azure/vSphere＋スナップショット | 技術者・自分 |

---

### Stage 1：現在（ローカル・Dockerのみ）

ローカル完結。追加コストなし。

- WSL上でファイル操作・コマンド実行（OSレベルの設定変更・アプリインストールは対象外）
- **Docker**：クリーンなLinuxコンテナ内でテスト・デバッグが可能
- Windows版はDockerなし・コード生成〜ドキュメントまで
- **限界**：Windowsアプリのインストール・OSレベル設定変更・RHEL固有操作などは対象外

---

### Stage 2：登録済みOS環境へのリモート接続

ユーザーが事前に環境を用意して登録しておく方式。

**フロー：**
```
ユーザーがUIで環境を登録（IP・認証情報・OS種別）
  例）Windows Server 2022  192.168.1.10  admin/****
      RHEL 9               192.168.1.20  root/****

ディスパッチャーがタスクを読んで判断
  └── 「Dockerでは無理」→ 登録済み環境を使用するよう指示
  └── 環境が登録されていない → 「環境を用意してください」と案内して終了

インフラAI・デバッグAIがSSH等でリモート接続して作業
```

**課題：環境汚染問題**
- インストールや設定変更が蓄積して次回以降のテストに影響する
- Stage 2 ではこの問題は未解決（利用者が管理する前提）

---

### Stage 3：スナップショット付きクリーン環境（最終段階）

常にフレッシュな状態を保証する完全自動フロー。

**プラットフォーム選択（UI で設定）：**
- Azure VM
- vSphere（既存Ansibleプレイブック活用）

**動作フロー：**
```
ユーザー「Windowsで動く〇〇を作って」
  │
  ├── 設計AI        → 設計書を書く
  ├── コーディングAI → コードを書く
  │
  ├── インフラAI    → 登録済みプラットフォーム（Azure/vSphere）に
  │                   スナップショットから復元してクリーン環境を用意
  │                   「環境準備完了・IPは〇〇」をステータスファイルに記録
  │
  ├── デバッグAI    → 準備された環境にSSH等で接続
  │                   実際にインストール・動作確認
  │                   「成功/失敗・ログ」をテスト結果ファイルに記録
  │
  └── ディスパッチャー → 完了判定 or 差し戻し
                         完了後：スナップショットへ自動ロールバック
```

**スナップショット運用：**
- テスト・デバッグ実行前に保存済みスナップショットから復元
- テスト・デバッグ完了後に自動ロールバック
- 常にフレッシュな状態を保証

#### vSphere のスナップショット（シンプル）

vSphere はスナップショット取得・リバートがネイティブ機能として組み込まれているため、`community.vmware` モジュールから数行で完結する。

```yaml
# リバート（テスト前）
- community.vmware.vmware_guest_snapshot:
    name: "{{ vm_name }}"
    snapshot_name: "clean-state"
    state: revert

# 取得（初回のみ）
- community.vmware.vmware_guest_snapshot:
    name: "{{ vm_name }}"
    snapshot_name: "clean-state"
    state: present
```

#### Azure のスナップショット（要工夫）

Azure はVM単位のスナップショットリバートがネイティブにないため、以下のフローで実現する。  
実装は `spec2325705/bk-yuichi.matsuo` リポジトリの `3Ti-NC-Role-FIX/rsrenew-snap_to_vm.yml` を改変して使用する。

```
【初回のみ】
  SYSPREP済みVMのOSディスクからスナップショット取得・保存

【テストのたびに】
  VM停止（deallocate）
  → VM削除（NIC・NSG・IPは維持）
  → 古いOSディスク削除
  → スナップショットから新しいOSディスク作成
  → 既存NIC・NSGを使って同名VMを再デプロイ
```

既存コードから不要な部分（NSG/NIC作成・毎回スナップショット作成）を削ることでシンプルになる。  
`vm_from_disk_template.json`（ARMテンプレート）はOS問わず流用可能。

ユーザーがやることは要件を言うだけ。設計・実装・環境構築・テストが全自動で完結する。

### 環境未登録時のUX方針：「使えない」で終わらない

```
「Dockerでは無理」とディスパッチャーが判断 & 登録済み環境なし
  │
  ├── 「テスト用に別の環境を用意できますか？（IPと認証情報を登録）」
  │     YES → セットアップ画面で登録 → テストまで自動
  │     NO  → 「Azure / vSphere アカウントはありますか？」
  │             YES → Stage 3 の設定を案内
  │             NO  → 「WSLとDockerで対応できる範囲での実装に切り替えます」
  │
  └── 社内・配下ユーザー向け
        → 構成済みUbuntuイメージ（Ansible・Docker入り）を適用するだけで即フルスタック
```

**原則：「この環境では使えません」だけで終わらせない。必ず次のアクションを示す。**

### WSL版の動作要件

WSL版はUbuntu前提で作られているため、WSLのディストリビューションは **Ubuntu（20.04 / 22.04 / 24.04）に限定する**。
Debian・Fedora・openSUSEでは動作しない部分がある（apt・systemd・bubblewrap等）。

  - ディレクター：ユーザーの意図を深く理解し「何を・なぜ・どこまで」を決める。品質基準・完了条件・優先順位を定義。実装詳細には踏み込まない。
  - ディスパッチャー：ディレクターの方針を受けて「誰が・何を・いつまでに」に落とす。タスク割り当て・進捗監視・催促。
  - 分ける価値がある場面：タスクが複雑で戦略的判断と実務調整が両方重い場合。一つにまとめると判断の質が下がる可能性。
  - 分けない方が良い場面：タスクがシンプルで「何を作るか」が自明な場合。トークンと複雑さが増えるだけ。
  - 方針：最初はディスパッチャーに統合し、必要を感じたら分離する。
