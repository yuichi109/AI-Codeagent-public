## 3. フック（hooks）⬜ 残り（優先度低）

**現状:** lint/test/フォーマットの実行はモデルの判断任せ。再現性がない。

**Claude Code の方式:** PreToolUse / PostToolUse 等で任意スクリプトを宣言的に差し込む（「編集後に必ず lint」など）。

**改善方向:**
- `config/hooks.json`（仮）でツール実行前後のコマンドを定義
- 例: write_file/edit_file の後に対象ファイルへ ruff/eslint を自動実行

---

## 4. 権限の粒度（allowlist）

**現状:** 承認は「全部出す」か「bypass で全部消す」かの二択に近い。

**問題:** 安全と自律のトレードオフが両極端。

**Claude Code の方式:** ツール × コマンドパターン単位の allowlist / denylist、permission mode。

**改善方向:**
- `run_command` のコマンド別ポリシー（`npm`/`git` は自動許可、`rm` は確認、など）
- 承認設定の永続化（セッションをまたいで「このコマンドは今後許可」）

---

## 5. 検証ループ（verify）✅ 完了（v1.9.0）

**実装内容:**
- `write_file` / `edit_file` の保存時に**自動構文チェック**を実行し、結果をモデルへ突き返す（[server.py:2889](../server.py)）
- チェッカー: `py_compile`（Python）/ `json` / `yaml` / `bash -n` / `node --check`。advisory 方式で合否を結果に注入
- 構文エラーがあればモデルが自己修正ループに入る

**当初の課題（解消済み）:** `code_lint` はあったが、変更後に自動で検証→自己修正する仕組みがなかった。

**残り（任意の発展）:** 実起動・ブラウザ検証（Playwright MCP）まで含めた E2E 検証ループは未統合。構文チェックレベルでは完了。

---

## 6. 根本のモデル性能 ⬜ 残り（次に効く）

**現状:** gpt-4.1-mini / gpt-5.4 系がデフォルト。

**本質:** agentic な粘り強さ・ツール選択の的確さ・diff の正確さは**モデルの地力**が支配的。足場を改善しても、ここで最終的な差が出る。

**改善方向:**
- 重要タスクは上位モデル（gpt-4.1 / gpt-5 系フル）に切り替えられる導線を強化
- Claude 系プロバイダーの追加検討（Anthropic API 対応）
