## 1. エージェントループの持久力（最重要）

**現状の制約:**
- `server.py`: `max_iterations = 30` — ツール呼び出し30回で打ち切り（[server.py:2024](../server.py)）
- `server.py`: `MAX_HISTORY_MESSAGES = 20` — 会話履歴を直近20件に**単純トリム**（[server.py:1593](../server.py)）

**問題:**
- 大規模リファクタや多ファイル実装は30回で力尽きる
- 単純切り捨てのため、長いタスクで**前半の文脈が消える** → 「何をしていたか」を見失う

**Claude Code の方式:**
- 数百ステップ回せる
- 自動 compaction（古い文脈を要約して保持。単純切り捨てしない）

**改善方向:**
- `max_iterations` を動的化（タスク規模・進捗で延長 / ユーザーが延長承認）
- 履歴トリムを要約ベースに（既存の `_recent_head_unsafe` / `SUMMARY_KEEP_RECENT` を活用・拡張）

## 2. 計画モード（Plan mode）

**現状:** 「即実装」か「承認モーダル」の二択。実装前に方針をすり合わせる仕組みがない。

**問題:** 大きな変更で走り出してから手戻りする。

**Claude Code の方式:** 探索 → 計画提示 → 承認 → 実装、と段階分離。計画中は書き込み禁止。

**改善方向:**
- UI に「計画モード」トグル追加（bypass トグルの隣）
- 計画モード中は書き込み系ツールを無効化し、最後に計画を提示して承認を取る
- マルチエージェントの「計画確認フロー」を単一AIにも展開できる

---

## 3. フック（hooks）

**現状:** lint/test/フォーマットの実行はモデルの判断任せ。再現性がない。

**Claude Code の方式:** PreToolUse / PostToolUse 等で任意スクリプトを宣言的に差し込む（「編集後に必ず lint」など）。

**改善方向:**
- `config/hooks.json`（仮）でツール実行前後のコマンドを定義
- 例: write_file/edit_file の後に対象ファイルへ ruff/eslint を自動実行

---

## 4. 権限の粒度（allowlist）

**現状:** 承認は「全部出す」か「bypass で全部消す」かの二択に近い。

**問題:** 安全と自律のトレードオフが両極端。

**Claude Code の方式:** ツール × コマンドパターン単位の allowlist / denylist、permission mode。

**改善方向:**
- `run_command` のコマンド別ポリシー（`npm`/`git` は自動許可、`rm` は確認、など）
- 承認設定の永続化（セッションをまたいで「このコマンドは今後許可」）

---

## 5. 検証ループ（verify）

**現状:** `code_lint` はあるが、変更→実起動→エラー確認→自己修正の自動ループがない。

**Claude Code の方式:** 変更後にアプリを起動・ブラウザ検証し、失敗を読んで直すまでやり切る。

**改善方向:**
- 「変更したら起動して確認」を促すプロンプト規律＋自動テスト実行ツール
- preview/ブラウザ検証の統合（既存 Playwright MCP を検証ループに組み込む）

---

## 6. 根本のモデル性能

**現状:** gpt-4.1-mini / gpt-5.4 系がデフォルト。

**本質:** agentic な粘り強さ・ツール選択の的確さ・diff の正確さは**モデルの地力**が支配的。足場を改善しても、ここで最終的な差が出る。

**改善方向:**
- 重要タスクは上位モデル（gpt-4.1 / gpt-5 系フル）に切り替えられる導線を強化
- Claude 系プロバイダーの追加検討（Anthropic API 対応）
